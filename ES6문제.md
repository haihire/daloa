1. 데코레이터 메타데이터 불안정성 (Reflect-metadata 문제)
NestJS는 @Injectable(), @Controller() 같은 데코레이터를 읽어와서 의존성 주입(DI)을 처리합니다. 이때 reflect-metadata라는 라이브러리를 사용하는데, ESM 환경에서 두 가지 결정적인 문제가 생깁니다.
실행 시점의 차이: CommonJS(CJS)는 코드가 실행될 때 동적으로 메타데이터를 등록하기 쉽지만, ESM은 코드가 실행되기 전 단계에서 모듈의 구조를 먼저 분석(Static Analysis)합니다. 이 과정에서 데코레이터가 생성하는 메타데이터가 아직 준비되지 않거나, 순서가 꼬여 NestJS가 "이 클래스가 어떤 타입인지 모르겠다"며 의존성 주입에 실패하는 경우가 잦습니다.
순환 참조(Circular Dependency) 취약성: ESM은 CJS보다 순환 참조에 훨씬 엄격합니다. NestJS 서비스끼리 서로 참조할 때, CJS에서는 대충 넘어갔던 것들이 ESM에서는 ReferenceError: Cannot access '...' before initialization을 내뱉으며 서버가 죽어버립니다.
2. 의존성 충돌 (CJS 전용 라이브러리와의 불화)
Node.js에서 ESM 모듈은 CJS 모듈을 import 할 수 있지만, 그 반대나 특정 상황에서는 에러가 발생합니다.
Named Export 미지원: 많은 구형 NestJS 라이브러리(@nestjs/schedule, passport 관련 모듈 등)는 module.exports = ... 방식을 씁니다. ESM에서 이를 import { TaskService } from ... 처럼 이름으로 가져오려 하면 "그런 거 없다"는 에러가 납니다. 오직 import pkg from '...' 처럼 전체를 가져와서 꺼내 써야 하는 불편함이 생깁니다.
__dirname, __filename 부재: NestJS 내부나 라이브러리에서 파일 경로를 찾을 때 흔히 쓰는 __dirname 변수는 ESM에 존재하지 않습니다. 이를 사용하는 오래된 라이브러리들은 ESM 환경에 로드되는 순간 바로 에러를 일으킵니다.
이중 인스턴스(Dual Package Hazard): 동일한 라이브러리가 CJS와 ESM 버전을 둘 다 가지고 있을 때, NestJS 내부에서 CJS 버전을 쓰고 개발자가 ESM 버전을 import 하면, 메모리에 두 개의 서로 다른 라이브러리 인스턴스가 올라갑니다. 이로 인해 싱글톤으로 동작해야 할 서비스가 두 개가 되어버려 데이터 동기화가 깨지는 대참사가 벌어집니다.
요약: 왜 불안정한가?
결국 "NestJS의 심장(데코레이터/DI)은 CJS의 유연함에 최적화되어 설계되었는데, ESM의 엄격한 규칙을 억지로 적용하려니 여기저기서 삐걱거리는 것"입니다.
이 때문에 NestJS 팀도 v12에서 컴파일러(SWC)와 핵심 엔진 자체를 ESM 표준에 맞게 갈아엎는 작업을 진행하고 있는 것입니다.

기술적 호기심으로 접근하신다면, 이 문제는 자바스크립트 생태계의 "거대한 과도기(CommonJS → ESM)"를 가장 잘 보여주는 사례입니다.
단순히 "안 된다"를 넘어, 내부적으로 어떤 설계적 충돌이 있는지 3가지 핵심 포인트로 더 깊게 파헤쳐 보겠습니다.
1. Reflect-metadata와 emitDecoratorMetadata의 한계
NestJS의 마법은 @Injectable()을 붙였을 때 생성되는 런타임 타입 정보에서 나옵니다.
CJS 방식: 컴파일러(tsc)가 코드를 변환할 때, 클래스 생성자의 파라미터 타입을 메타데이터로 저장합니다. CJS는 실행 시점에 이 정보를 동적으로 읽어오기 매우 유연합니다.
ESM의 엄격함: ESM은 모듈을 로드할 때 정적 분석(Static Analysis)을 강제합니다. 실행하기 전에 모든 관계가 정의되어야 하죠. 그런데 데코레이터가 메타데이터를 주입하는 시점과 ESM이 모듈을 완전히 확정 짓는 시점이 미세하게 어긋나면서, NestJS가 "이 클래스에 주입할 서비스가 대체 무슨 타입이지?"라고 물었을 때 undefined를 반환하는 상황이 발생합니다.
2. 순환 참조(Circular Dependency) 처리의 철학 차이
NestJS 앱이 커지면 A 서비스가 B를 참조하고, B가 다시 A를 참조하는 경우가 생깁니다.
CJS (관대함): require는 일단 빈 객체({})라도 반환해서 실행을 이어갑니다. 나중에 값이 채워지면 운 좋게 돌아가죠.
ESM (엄격함): ESM은 "완성되지 않은 모듈"을 허용하지 않습니다. 참조가 꼬여 있으면 즉시 ReferenceError를 띄우며 프로세스를 종료시킵니다. NestJS의 DI 컨테이너는 CJS의 유연한 구조 위에 설계되었기 때문에, ESM의 이 엄격함을 견디지 못하고 터져버리는 것입니다.
3. "Dual Package Hazard" (이중 인스턴스 문제)
가장 골치 아픈 문제입니다. NestJS는 많은 외부 라이브러리에 의존합니다.
만약 어떤 라이브러리(예: class-transformer)가 CJS와 ESM 버전을 둘 다 제공한다고 가정해 봅시다.
NestJS 코어는 내부적으로 CJS 버전을 로드하고, 여러분의 프로젝트는 "type": "module" 설정 때문에 ESM 버전을 로드합니다.
결과적으로 메모리에 똑같은 라이브러리 객체가 두 개 생깁니다. instanceof 체크가 실패하고, 한쪽에서 설정한 전역 옵션이 다른 쪽에는 적용되지 않는 기괴한 버그가 발생합니다.
💡 결론적으로
NestJS 개발진이 v12에서 "Native ESM"을 선언한 것은 단순히 유행을 따르는 게 아니라, "이제는 CJS의 편법으로 ESM의 엄격함을 속이는 데 한계가 왔다"고 판단했기 때문입니다.
이를 해결하기 위해 tsc를 버리고 SWC(Rust 기반 컴파일러)를 전면 도입하여, ESM 환경에서도 메타데이터가 정확한 타이밍에 주입되도록 엔진 자체를 재설계하고 있는 것입니다.
